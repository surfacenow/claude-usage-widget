# 開発ノート: Claude Usage Widget UI改善 (2026-03-19)

新人エンジニア向けのリファレンスとして、この日の作業で何が起きたか・何に詰まったか・どう解決したかを記録する。

---

## 前提: このウィジェットの仕組み

デスクトップ上に常駐する小さなオーバーレイ。Claude API の使用量をバー表示する。

- **Tkinter** (Python標準のGUIライブラリ) で描画
- `-transparentcolor` で背景を透明にして、デスクトップに浮かせている
- 右クリックメニューからテーマ切替・手動スキャンなどができる

ウィジェットのサイズは **236x32ピクセル** しかない。この極小サイズが、今回のほぼ全ての問題の根本原因。

---

## 1. 文字が潰れて読めない問題

### 何が起きたか

ライトモード・ダークモード両方で、文字がにじんでぼやけていた。
隣のWindows時計(「19:36」とか)はくっきりなのに、ウィジェットの文字だけ汚い。

### 原因は3つあった

#### (a) DPI Awareness が未設定

Windows は高解像度ディスプレイ向けにDPIスケーリングをかける。
アプリが「自分はDPI対応してます」と宣言しないと、Windowsが勝手に拡大してぼやける。

```python
# 修正: プログラム起動時に宣言する
try:
    ctypes.windll.shcore.SetProcessDpiAwareness(1)
except Exception:
    ctypes.windll.user32.SetProcessDPIAware()
```

**学び**: GUIアプリを作るとき、Windowsでは最初にDPI Awarenessを設定するのが定石。やらないと「なんかぼやける」という原因不明のバグに見える。

#### (b) フォントが違った

元は `Segoe UI` を使っていたが、Windows 11 のシステムUIは `Segoe UI Variable` を使っている。
同じ Segoe ファミリーでも、Variable版の方がレンダリングが綺麗。

```python
# Before
font_main = ("Segoe UI", 9, "bold")

# After
font_main = ("Segoe UI Variable", -12)
```

ちなみに **フォントサイズを負の値にするとピクセル単位指定**になる。
Tkinterのデフォルトはポイント単位で、DPIによって見た目が変わってしまう。ピクセル指定の方が予測しやすい。

#### (c) bold が邪魔

小さいサイズで bold にすると、隣のピクセルと潰れて読みにくくなる。
Windows時計のフォントは bold ではない。細い方がくっきり見える。

**学び**: 「読みにくい → bold にしよう」は直感的だけど、小さいUIでは逆効果。

## 2. ライトモードの配色問題

### 何が起きたか

もともとダークモード前提で作った配色だった。
シアン (`#55DDFF`) やグレー (`#999999`) は暗い背景では映えるが、明るい背景では完全に溶ける。

### どう考えたか

- ダークモード: 元のシアン系をそのまま維持（これは問題なかった）
- ライトモード: Claude のブランドカラーであるテラコッタ/ブラウン系に全面変更

```python
# ライトモードのアクセント色
"accent": "#A0522D",   # テラコッタブラウン
"muted": "#3D2B1F",    # 濃いブラウン（ラベル用）
"reset_text": "#3D2B1F",
```

### テキストシャドウの罠

ダークモードでは文字の後ろに1pxずらした影ラベルを置いて、浮き出し効果を出していた。
ライトモードでこれをやると、文字の周りがぼやけて二重に見える。

```python
# 影ラベルは本体の1px右下に配置される
self.rst_5h_shadow.place(x=rst_x + 1, y=y0 + 1)
self.rst_5h.place(x=rst_x, y=y0)  # 本体
```

解決策: ライトモードのシャドウ色を透明色 (`#010101`) と同じにして、事実上無効化した。

**学び**: ダークUIで有効なテクニックがライトUIでは逆効果になることがある。テーマ対応は色を変えるだけじゃなく、エフェクトの有無も切り替える必要がある。

## 3. グリップ(ドラッグハンドル)のデザイン

### 背景知識: transparentcolor の制約

Tkinterの `-transparentcolor` は指定した色のピクセルをクリック透過にする。
つまり**透明な部分はマウスで掴めない**。

グリップを6つのドットだけで構成すると、ドットの間の隙間は透明 = クリックできない。
小さいドットだと「掴もうとしても掴めない」という UX問題が起きる。

### 調整したパラメータ

```python
dot_r = 2.0           # ドット半径
for dx in (5, 13):    # 横位置 (2列)
    for dy in (7, 14, 21):  # 縦位置 (3行)
```

ポイント:
- ドットの**サイズ**と**中心間距離**はセットで調整する
- 片方だけ変えると見た目のバランスが崩れる
- 見本画像を拡大して補助線を引くと、ズレが明確になる

**学び**: 数ピクセルの世界では、1-2pxのズレが肉眼でわかる。「だいたい合ってるはず」は通用しない。実際にスクショを撮って比較する。

## 4. システムテーマ自動追従 — 一番苦戦した部分

### やりたかったこと

Windows の 設定 > 個人用設定 > 色 でダーク/ライトを切り替えたら、ウィジェットも自動で追従する。

### 試行1: ポーリング (動いたけどイマイチ)

3秒ごとにレジストリを読みに行く方式。

```python
def _poll_system_theme(self):
    if self.theme_mode == THEME_AUTO:
        sys_theme = get_system_theme()  # レジストリ読み取り
        if sys_theme != self.theme_name:
            # テーマ変更
    self.root.after(3000, self._poll_system_theme)
```

レジストリの場所:
```
HKEY_CURRENT_USER\SOFTWARE\Microsoft\Windows\CurrentVersion\Themes\Personalize
値: AppsUseLightTheme (1=ライト, 0=ダーク)
```

動くけど「変更がなくても3秒ごとに無駄に読みに行く」のが気持ち悪い。

### 試行2: WM_SETTINGCHANGE フック (失敗)

Windowsはシステム設定が変わると `WM_SETTINGCHANGE` メッセージをブロードキャストする。
これを受け取ればプッシュ型にできる。

```python
# ウィンドウプロシージャをサブクラス化して WM_SETTINGCHANGE を捕捉
user32.SetWindowLongPtrW(hwnd, GWL_WNDPROC, custom_wndproc)
```

**結果: 動かなかった。**

理由: このウィジェットは `overrideredirect(True)` で作られている（タイトルバーなし・タスクバーに出ない）。
この状態のウィンドウは、Windows のブロードキャストメッセージを受け取れない場合がある。

さらに `winfo_id()` と `root.frame()` で取得できるHWNDも試したが、どちらもダメだった。

**学び**: Tkinter + overrideredirect の組み合わせは、Windows のメッセージングと相性が悪い。「理論上できるはず」が通用しない領域。

### 試行3: RegNotifyChangeKeyValue (成功)

Windows API に「レジストリキーが変わるまでブロックする」関数がある。

```python
def _watch_system_theme(self):
    key = winreg.OpenKey(
        winreg.HKEY_CURRENT_USER, REG_PATH, 0,
        winreg.KEY_NOTIFY | winreg.KEY_READ,
    )

    def _watcher():
        while True:
            # この呼び出しはレジストリが変わるまで止まる(ブロッキング)
            ctypes.windll.advapi32.RegNotifyChangeKeyValue(
                int(key), False, REG_NOTIFY_CHANGE_LAST_SET, None, False,
            )
            # 変わった！メインスレッドに通知
            self.root.after_idle(self._on_system_theme_changed)

    t = threading.Thread(target=_watcher, daemon=True)
    t.start()
```

ポイント:
- `RegNotifyChangeKeyValue` は**変更があるまで戻ってこない** → CPU使用率ゼロ
- ブロッキングなので**別スレッドで回す**必要がある
- GUIの更新は `root.after_idle()` でメインスレッドに渡す（Tkinterはスレッドセーフじゃない）
- スレッドは `daemon=True` にして、メインが終了したら一緒に死ぬようにする

**学び**:
- 「プッシュ通知」の実現方法は1つじゃない。OS のメッセージング、レジストリ監視、ファイル監視など複数の手段がある
- 1つの方法が環境の制約でダメでも、別のアプローチで同じ結果を得られることが多い
- ブロッキング API + スレッド は、ポーリングよりエレガントで効率的

## まとめ: この日の教訓

1. **小さいUIは難しい** — 数ピクセルの差が見た目を大きく左右する。スクショ拡大で確認する癖をつける
2. **プラットフォーム固有の初期設定を忘れない** — DPI Awareness のように、やらないと原因不明のバグに見えるものがある
3. **ダークUIとライトUIは別物** — 色を反転させるだけでは済まない。エフェクトの有効/無効も含めて設計する
4. **理論と実践は違う** — WM_SETTINGCHANGE は「仕様上は全ウィンドウに届く」が、実際には overrideredirect で届かなかった
5. **詰まったらアプローチを変える** — 同じ穴を掘り続けるより、別の入口を探した方が早い
